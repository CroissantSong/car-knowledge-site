# 岗位与协作

## 研发组织架构

典型的主机厂研发中心组织：

| 板块 | 英文 | 核心职责 |
|------|------|----------|
| 造型 | Design / Studio | 外饰、内饰、CMF、A 面数据 |
| 工程 | Engineering | 车身、底盘、动力、三电、电子电气、智能化系统 |
| 试制 | Prototype | 样车制造、快速样件、试制问题闭环 |
| 试验 | Test & Validation | 台架、整车、碰撞、三高、EMC 等验证 |
| 项目管理 | PM | 节点、资源、风险、成本和跨部门推进 |

### 造型（Design/Studio）

- **外饰设计师**：整车外观造型
- **内饰设计师**：仪表台、座椅、门板设计
- **CMF 设计师**：Color（颜色）、Material（材料）、Finishing（表面处理）
- **数字模型师**：Alias/ICEM 曲面建模（A面）

### 工程（Engineering）

| 部门 | 负责系统 |
|------|----------|
| 车身工程 | 白车身、开闭件（车门/引擎盖/尾门） |
| 底盘工程 | 悬架、制动、转向、轮胎 |
| 动力总成 | 发动机、变速箱、混动系统 |
| 三电（新能源） | 电池、电机、电控 |
| 电子电气 | 线束、ECU、网络架构 |
| 智能驾驶 | 感知、决策、控制 |
| 智能座舱 | 中控/仪表、HMI、语音 |
| NVH | 噪音/振动/声振粗糙度 |
| 安全 | 碰撞安全、行人保护 |

### 试制（Prototype）

- VP/TT/PP 样车制造
- 试制车间管理
- 快速样件/3D 打印

### 试验（Test & Validation）

- **台架试验**：发动机台架、底盘测功机
- **整车试验**：三高（高温/高寒/高原）、耐久路试
- **碰撞试验**：正碰/侧碰/偏置碰
- **EMC 试验**：电磁兼容性

## 供应链角色

| 层级 | 面向对象 | 典型交付 |
|------|----------|----------|
| OEM | 用户和整车项目 | 产品定义、整车集成、品牌、制造和销售 |
| Tier 1 | OEM | 系统或模块级方案，如制动系统、座椅、电池包 |
| Tier 2 | Tier 1 或 OEM | 关键零部件，如传感器、活塞、连接器 |
| Tier 3 | 上游供应链 | 原材料、基础件、芯片裸片等 |

| 层级 | 角色 | 提供 | 示例 |
|------|------|------|------|
| **OEM** | 主机厂 | 整车设计、总装、品牌 | 比亚迪、吉利、蔚来 |
| **Tier 1** | 一级供应商 | 系统/模块 | 博世（ESP/ECU）、宁德时代（电池） |
| **Tier 2** | 二级供应商 | 零部件 | 活塞供应商、传感器供应商 |
| **Tier 3** | 三级供应商 | 原材料/基础件 | 钢材、芯片裸片 |

## 项目管理与节点

### 里程碑（Milestone）

与 GVDP 关键节点对齐，每个里程碑都有明确的准入/准出标准。

### 阀点（Gateway）

重大决策评审点，通常需要高层审批才能通过。

> 阀点没过 → 项目不能进入下一阶段，类似于「门关着，必须把问题解决才能通过」。

### 核心交付物

每个节点需要提交一系列文档和数据：

- 设计数据（CAD 3D/2D）
- BOM（物料清单）
- DVP 报告（设计验证计划结果）
- DFMEA/PFMEA
- 成本分析
- 质量目标达成情况

### 项目经理（PM）

- 协调跨部门资源
- 跟踪项目进度和节点
- 管理风险和问题
- 向上汇报项目状态

## 新员工入门建议

1. **熟悉 GVDP**：了解整车开发从 0 到 1 的全过程
2. **掌握核心缩写**：SOP/EOP/DVP/BOM/PPAP 等是最常用的术语
3. **理解上下游**：你的工作交付给谁？接收谁的工作？
4. **多看实车**：拆车报告、对标分析是最好的学习材料

## 场景化学习卡

### 1. 研发组织与岗位协作

**场景问题：** 一个简单的制动问题，为什么会有底盘、电子、软件、试验、质量一起开会？

**简要结构图：**

| 角色 | 在问题闭环中的作用 |
|------|--------------------|
| 系统工程 | 判断问题归属、接口和整车影响 |
| 零部件工程 | 检查硬件设计、供应商件和装配状态 |
| 软件/标定 | 检查控制逻辑、参数和版本 |
| 试验/质量/售后 | 提供场景证据、批次追溯和关闭标准 |

**原理（说人话）：** 车企按专业分工，但车辆问题按系统耦合发生。底盘工程关注硬件和性能，电子软件关注信号与逻辑，试验提供证据，质量追踪批次和关闭，售后反馈真实用户场景。协作的关键是把现象、数据、责任和关闭标准放到同一张桌上。

**对比/类比：** 像医院会诊：病人只有一个症状，但可能需要内科、影像、检验和护理共同判断。

**车企工作场景：** 新人要学会在会议中说清“我负责哪段链路、已有证据是什么、下一步需要谁支持”，而不是只说“可能是某某问题”。

**小测：** 如果用户反馈 AEB 误触发，至少需要哪些专业一起看？

### 2. 供应链角色

**场景问题：** 为什么车企不把每个零件都自己做？

**简要结构图：**

| 层级 | 交付粒度 | 新人要关注 |
|------|----------|------------|
| OEM 整车厂 | 整车目标和系统集成 | 需求、接口、验证和品牌责任 |
| Tier 1 | 系统/总成 | 性能、成本、样件节奏和质量证据 |
| Tier 2 | 关键零部件 | 尺寸、材料、可靠性和供货稳定性 |
| Tier 3 | 材料/基础件 | 原材料波动、产能和合规风险 |

**原理（说人话）：** 整车由成千上万零件组成，供应链负责把专业能力、成本和产能组织起来。OEM 定义整车目标并集成验证；Tier 1 常交付系统级方案，如座椅、制动、智驾硬件；Tier 2/3 提供更基础的零件和材料。

**对比/类比：** 像装修房子：业主定目标，总包统筹，水电、木工、材料商各有专业分工。

**车企工作场景：** 供应商问题不能只看“谁做坏了”。还要看需求是否清楚、接口是否稳定、样件是否按节奏、变更是否同步、质量证据是否充分。

**小测：** 如果一个雷达支架尺寸偏差导致标定失败，OEM 和供应商分别要承担哪些排查动作？

### 3. 项目管理与节点交付

**场景问题：** 工程师找到更好的技术方案，为什么项目经理还会问成本、节点和交付物？

**简要结构图：**

| 约束 | 项目管理要问的问题 |
|------|--------------------|
| 技术方案 | 性能提升是否真实解决用户问题 |
| 成本 | 单车成本、开发费、模具费是否可接受 |
| 质量 | 是否有充分验证和量产稳定性证据 |
| 时间 | 是否影响关键节点和供应链节奏 |
| 用户价值 | 用户是否愿意为改动买单 |

**原理（说人话）：** 量产车不是技术展示品，而是在目标价格、目标时间和目标质量下交付用户价值。一个方案性能更好，可能带来单车成本上升、供应风险、验证周期延长或售后复杂度增加。项目管理的工作是让取舍透明，并推动跨团队按节点兑现承诺。

**对比/类比：** 像做毕业设计和量产商品的差别：前者可以追求最好效果，后者还要保证每一件都稳定、可买、可修、按时交付。

**车企工作场景：** 评审材料常要求写清 BOM 成本、开发费、模具费、重量、风险、验证计划和收益。新人要能把技术建议翻译成项目语言。

**小测：** 为什么“更高算力芯片”不一定是智驾项目的最优方案？
